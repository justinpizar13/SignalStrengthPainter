# WiFi Buddy — marketing site

Static marketing site for the WiFi Buddy iOS app. Built with [Astro 5](https://astro.build/) + Tailwind v4 + MDX.

## What's here

```
website/
├── astro.config.mjs            ← site URL, integrations
├── public/
│   ├── favicon.svg             ← brand mark (mirrors AppLogoView.swift)
│   ├── og-default.svg          ← default Open Graph card
│   └── robots.txt              ← allow-list incl. AI crawlers
├── scripts/
│   └── sync-legal.mjs          ← copies PrivacyPolicy.md / TermsOfUse.md
│                                  from the repo root into src/content/legal
├── src/
│   ├── components/             ← Logo, Header, Footer, FeatureCard, …
│   ├── content/
│   │   ├── blog/*.mdx          ← evergreen posts
│   │   └── legal/*.md          ← synced from repo root (do not edit here)
│   ├── content.config.ts       ← Astro content collections
│   ├── layouts/                ← BaseLayout (SEO + JSON-LD), PostLayout
│   ├── lib/site.ts             ← canonical site metadata (URL, App Store
│   │                              link, pricing) — single source of truth
│   ├── pages/                  ← / · /features · /devices · /vs-* · /blog
│   │                              · /blog/[...slug] · /privacy · /terms
│   └── styles/global.css       ← brand tokens + Tailwind v4 import
└── package.json
```

## Run it

```bash
npm install
npm run dev          # http://localhost:4321
npm run build        # static output in dist/
npm run preview      # preview the production build locally
```

`predev` and `prebuild` automatically run `scripts/sync-legal.mjs`, which copies the canonical legal markdown from the iOS repo into `src/content/legal/`. Don't hand-edit the files in `src/content/legal/` — edit the originals at the repo root.

## Before launching

A few one-time things to update before flipping the DNS:

1. **App Store URL** — `src/lib/site.ts → SITE.appStoreUrl`. Replace `idTODO` with your real App Store listing ID once it's live.
2. **Domain** — `astro.config.mjs → site` and `src/lib/site.ts → SITE.url`. Both should point at your real production domain.
3. **Aggregate rating** — `src/lib/site.ts → SITE.rating`. Fill in once App Store Connect has real numbers, or set to `null` to omit the `aggregateRating` JSON-LD until then.
4. **Smart App Banner** — uncomment the `apple-itunes-app` meta tag in `src/layouts/BaseLayout.astro` and substitute the App Store ID. This makes Mobile Safari show an "Open in App Store" banner above the page.
5. **Hero rating numbers** in copy — search `4.8` if you don't want to surface a hard-coded rating before reviews land.

## Deploy

The site is plain static HTML — every host that serves files works:

- **Cloudflare Pages** — `npm run build`, point at `dist/`. Recommended; free, fast, on the same network as `speed.cloudflare.com` if you ever want to dogfood your own speed test against the marketing site.
- **Vercel** — Astro adapter is auto-detected.
- **Netlify** — same.
- **GitHub Pages** — `dist/` is publishable, but you'll need to override the `site:` URL or set up a custom domain.
- **S3 + CloudFront** — drop `dist/` on S3, point CloudFront at it, set `index.html` as the default root + `404.html` as the error doc.

After deploy, submit the sitemap to:

- [Google Search Console](https://search.google.com/search-console) — `https://wifibuddy.app/sitemap-index.xml`
- [Bing Webmaster Tools](https://www.bing.com/webmasters) — same URL

## Adding a blog post

Drop a new `.mdx` file in `src/content/blog/` with frontmatter:

```mdx
---
title: "Your post title"
description: "150-160 character search-snippet description."
pubDate: 2026-05-01
keywords: ["primary keyword", "secondary keyword"]
---

Markdown body. MDX components allowed.
```

The post is automatically picked up by:

- `/blog/index.astro` (sorted by `pubDate` descending, drafts hidden)
- `/blog/[...slug].astro` (renders the post body)
- `@astrojs/sitemap` (added to the sitemap)
- `Article` JSON-LD (auto-generated from the frontmatter)

The slug is the filename minus `.mdx`. Set `draft: true` in frontmatter to keep a post out of the index but still publish at its slug for review.

## Updating the legal docs

Don't edit `src/content/legal/*.md` directly. Edit the canonical files:

- `<repo-root>/PrivacyPolicy.md`
- `<repo-root>/SignalStrengthPainter/TermsOfUse.md`

`scripts/sync-legal.mjs` re-copies them on every `npm run dev` and `npm run build`. Both the iOS app's in-app `LegalDocumentView` and the public `/privacy` and `/terms` URLs serve the same text — App Store reviewers can audit one against the other.

## Brand tokens

The traffic-light + dark-gray palette mirrors `AppLogoView.swift` and `scripts/generate_app_icon.py` in the parent repo. The constants live in `src/styles/global.css` under the `@theme { ... }` block:

- `--color-bg` / `--color-bg-card` / `--color-border` — dark surface stack
- `--color-signal-green` `#4DD966` — outer Wi-Fi arc (excellent)
- `--color-signal-yellow` `#FAD138` — middle Wi-Fi arc (fair)
- `--color-signal-orange` `#FA8538` — inner Wi-Fi arc (poor)
- `--color-signal-red` `#FA5252` — center dot (dead zone)
- `--color-signal-blue` `#3B82F6` — interactive accent

If you change the iOS app's brand palette, update these tokens too so the site stays consistent.
