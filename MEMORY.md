# SignalStrengthPainter — session memory (reference for future chats)

Concise primer for future Cursor chats. The codebase is the source of truth; this doc covers the rationale, gotchas, and current-state pointers that aren't obvious from reading the code.

## Product goal

iOS SwiftUI app similar to **NetSpot**: walk a space, see a marker on a floor plan, and paint signal quality (TCP connect latency to `8.8.8.8:53`) onto the map as a heatmap overlay plus a breadcrumb path.

## Branding

App is **WiFi Buddy** (repo name remains `SignalStrengthPainter`). Premium tier is **WiFi Buddy Pro**. `CFBundleDisplayName = "WiFi Buddy"`. Bundle ID is `com.wifibuddy.app`.

The brand was originally "Wi-Fi Buddy" (hyphenated) but that name was taken on the App Store. The hyphen was dropped everywhere except Klaus's greeting (which still references "Wi-Fi" the *technology*).

### App Icon

Custom 1024×1024 PNG in `Assets.xcassets/AppIcon.appiconset/` (iOS 17+ universal format). Generated by `scripts/generate_app_icon.py` (Pillow) so the asset and the in-app `AppLogoView` stay visually identical.

**Current design — traffic-light Wi-Fi on dark gray:**
- Background `(40, 40, 44)` full-bleed (iOS applies its own squircle mask).
- Three concentric 90° fan arcs (225°→315°) + center dot. Each element a flat solid color: outer green `(77, 217, 102)`, middle yellow `(250, 209, 56)`, inner orange `(250, 133, 56)`, dot red `(250, 82, 82)`.
- Stroke `0.080`; arc bbox radii `0.445 / 0.315 / 0.185`; dot radius `0.055`; gaps `0.050`. `cy = 0.695` (vertically centered).
- No gradient, no sheen, no sparkles.
- Rendered at 4× supersampling, downsampled with LANCZOS to 1024×1024; flattened to opaque RGB (App Store requires no alpha).

`ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon` in `project.pbxproj`.

### In-App Branding (`AppLogoView.swift`)

Reusable SwiftUI `Canvas` view drawing the same Wi-Fi glyph on a **transparent background** (the dark-gray squircle is reserved for the home-screen PNG). Geometry constants match the icon PNG (`cy = 0.695`, radii `0.445 / 0.315 / 0.185`, stroke `0.080`, dot `0.055`, sweep 225°→315°). Four flat colors (`outerArcColor` / `middleArcColor` / `innerArcColor` / `dotColor`) — no `IridescentPalette`, no `colorScheme` dependency.

Accepts a `size` parameter; everything scales proportionally. Used in tab headers (Speed, Survey, Signal, Devices), `PaywallView` hero, and the assistant header.

## Tracking & positioning

### Why AR (not pedometer + heading)

Indoor pedometer + compass/yaw drifts and batches poorly for floor alignment. Moved to ARKit world tracking for smoother, 2D-faithful motion.

### World-space delta

Position is **`currentWorldPosition - originWorldPosition`** from `camera.transform.columns.3`, with `originWorldPosition` reset on `reanchorCurrentPose()`. Earlier code used only `inverse(origin) * current`'s translation column, which collapsed motion to a diagonal line.

### Map projection (`projectToMap` in `SignalMapViewModel`)

- Horizontal plane uses AR `x` and `z`. `pointsPerMeter = 38`.
- `localX = position.x * pointsPerMeter` (sign tuned so strafe matches expectation).
- `localY = position.z * pointsPerMeter`.
- Rotate by `storedMapRotationDegrees`.
- Anchor: `currentPosition = mapAnchorOffset + projectToMap(relativeAR)`.

### Alignment without a "which way is forward" step

1. **Start:** tap start → Start Survey (locks AR origin).
2. **Drift correction:** Re-anchor Here → tap known location.
3. **Optional rotation refinement:** on the **first** landmark re-anchor, if both segments are long enough, set `storedMapRotationDegrees` from the angle between map delta and AR horizontal delta.

Tunables: `minimumLandmarkSegmentPoints`, `minimumLandmarkARPixels`.

### Sample density

`minimumSampleDistancePoints = 16` (≈42 cm) and `minimumSampleInterval = 0.25 s` after April 2026 tuning. Heatmap blobs (108 pt ≈ 2.8 m) overlap ~85% at this spacing — still smooth — and tap-to-inspect targets stop fighting each other at default zoom.

## UI / UX flows

### Calibration stages (`CalibrationStage`)

`idle` → `selectingStartPoint` → `readyToStart` → `surveying` → `selectingReanchorPoint` → `finished`.

### Stop vs reset

- **Stop Survey** (`stopSurvey()`): stops AR + ping loop, sets `finished`, keeps trail and samples.
- **Reset**: clears everything; gated by `confirmationDialog`.

### Expanded map layout

`usesExpandedMapLayout` is true for `surveying` / `selectingReanchorPoint` / `finished`. Map-first; `mapContentScale = 0.72` baseline (was 0.48; bumped so longer walks fit).

### Map interaction

- **Pinch to zoom** — `MagnificationGesture` + `@GestureState pinchDelta`; clamped `[0.4, 5.0]` on commit. Effective scale = `contentScale * userScale * pinchDelta`.
- **Drag to pan** — `DragGesture(minimumDistance: 0)`. Drags ≤ `tapSlop (6 pt)` are treated as taps; longer drags update `panOffset`. Composed with pinch via `SimultaneousGesture`.
- **Zoom controls overlay (top-trailing)** — `+`, `-`, `recenter` buttons.
- **Tap-to-inspect** — when a tap (drag distance ≤ `tapSlop`) lands within `pointHitRadius / effectiveScale` of a `TrailPoint`, shows `PointInfoCard` (latency, capture time). Selected point gets a yellow waypoint + ring highlight. Recenter clears the selection.
- **Tap routing** — empty-space taps still forward to `onMapTap` for calibration stages, with the pan/zoom-corrected map coordinate.

### Post-survey insights (finished stage)

When `calibrationStage == .finished`, `ContentView` switches to `finishedLayout` (map at 300 pt height + insights panel). `SurveyInsightsEngine.generate` returns a `SurveyInsightsReport` rendered by `SurveyInsightsView`. `< 8` rated samples → `SurveyInsightsPlaceholder` instead.

Report contents: grade card, 2×2 stat grid, coverage mix bar, latency range strip (Best / Median / Worst 5% / Worst), ranked `[SurveyInsight]` cards.

Insights synthesized: coverage breakdown, dead-zone count + worst-zone stats (cluster threshold 1.8 m, min 2 points), latency profile (separately catches the **p95 > 200 ms with median ≤ 80 ms** spikes-but-mostly-fine case), stability warning (jitter/median ≥ 0.5 AND jitter ≥ 20 ms), router-direction hint (Pearson, requires ≥ 3 m walked and `|r| ≥ 0.35`), and a tailored "What to do next" recommendation.

Why deterministic and not LLM: same walk → same report; debuggable; no cloud calls; tunable in one file. Clean seam to swap or wrap later.

### Best/Worst spot markers

`SignalCanvasView` accepts optional `bestSpot` / `worstSpot` `TrailPoint`s. `drawHighlightedSpots` renders a 14-pt filled circle (green/red) with a 2.5 pt white border, an SF Symbol glyph (star/exclamation), and a "Best"/"Worst" capsule label offset above/below. Computed inline in `ContentView` via a `min`/`max` pass; only shown when `calibrationStage == .finished`. `SurveyInsightsView.latencyRange` renders a matching legend hint underneath the four tiles.

### Visual style (canvas)

- **Floor plan backdrop** — `drawFloorPlan` dispatches on `floorPlan.rooms`. Non-empty templates render rooms with `FloorPlanRoomTint`, faint "furniture" rectangles (only when room > 24 pt), and labels (only when room ≥ 48×28 pt). Honors user-supplied nicknames via `roomNameOverrides`.
- **Heatmap** — radial gradients per `TrailPoint` using `heatColor` from `SignalTrailModels`.
- **Path** — blue segments + waypoints; surveyor uses an SF Symbol via `Canvas` symbols.

- **StoreKit 2 integration (`ProStore.swift`):**
  - Product IDs: `com.wifibuddy.pro.sub.monthly` and `com.wifibuddy.pro.sub.yearly` (constants at top of file). The `.sub.` infix is permanent — Apple blocks reuse of deleted product IDs, and the original `.monthly` / `.yearly` IDs were deleted during an ASC subscription rebuild.

  - Detached `Transaction.updates` listener handles Ask-to-Buy / family-sharing / refund events.
  - `restore()` calls `AppStore.sync()` and re-derives entitlements.

To test buys on device, either keep Xcode attached, use the Simulator, or configure ASC products + Sandbox tester.

## Speed Test (download / upload)

`SpeedTestManager` runs a four-phase test: **Server Selection → Ping → Download → Upload**.

- **Server selection (added April 2026):** Fetches `https://speed.cloudflare.com/meta` (3 s timeout) for the colo code (e.g. `PHX`), client geolocation, and ISP (AS organization + ASN). Runs warmup latency probe in parallel. Maps colo via `CloudflareColoDirectory` (~80-entry static table covering all NA POPs + busiest global). Computes client→colo great-circle distance via `haversineMiles`. `isLikelySuboptimal` flags distance > **500 mi** (catches ISP routing failures like AZ → DFW ≈ 890 mi without flagging reasonable hops like AZ → LAX ≈ 370 mi). Reserves the first 0→1/4 of the progress bar.
- **Latency:** 10 HTTP RTTs to `speed.cloudflare.com/__down?bytes=0`; trimmed mean for ping, consecutive-difference mean for jitter.
- **Download:** 8 concurrent async streams in a `TaskGroup`. Each fetches 4 MB chunks via `URLSession.data(for:)` for the full phase duration. Cache-busting query param + `Accept-Encoding: identity` header. Only HTTP 200 counts. (Earlier delegate-based attempts: `URLSessionDataDelegate` callbacks never fired for Cloudflare's `__down` endpoint.)
- **Upload:** Same pattern — 8 concurrent streams POSTing 4 MB chunks via `URLSession.upload(for:from:)`. Bytes only count after server responds (HTTP 200), measuring true upload throughput rather than kernel buffer speed. (Earlier delegate-based attempts: `didSendBodyData` reported kernel-buffered bytes nearly instantly, so the phase appeared complete in ~1 s.)
- **Shared sampling loop** (`sampleTransferPhase()`): reads `TransferCounter` every ~250 ms, publishes a rolling 4-sample-window average as `currentSpeed`.
- **Final speed:** total bytes / total elapsed time (true average throughput).
- **Duration cap:** 12 s per phase.
- **Server info strip:** Provider + colo + city ("Cloudflare PHX · Phoenix, AZ"), distance ("~12 mi away"), ISP. Tints amber when `isLikelySuboptimal`.

UI: empty / active (phase indicator + live Mbps gauge + sparkline + progress bar) / complete (Download/Upload tiles + Ping/Jitter badges) plus a post-test Wi-Fi Report card.

### Klaus mascot

Pixel-art astronaut-robot mascot. Lives in `KlausMascotView.swift`. Replaces generic assistant branding so users feel like they're chatting with a character.

**Palette (current):** original artist source colors — cream body, light-blue TV-screen face, dark-blue eyes, pink cheeks, red belly, peach antenna tip, near-black outlines. (An earlier two-tone white + forest-green recolor was attempted at asset-generation time but the source pixel-art didn't separate cleanly on luminance/chroma; the recolor was dropped and `KlausMascotView.invertColors` defaults to `false`.)

**Asset pipeline:** `scripts/generate_klaus_frames.py` (Pillow) takes the source GIF and produces two shipped GIFs:

1. `KlausMascot.dataset/klaus.gif` (full body, 336×446) — every frame backdrop-knocked-out, cropped to a union bbox. Bbox computed on alpha (post-recolor), so dither noise is swallowed.
2. `KlausMascotHead.dataset/klaus_head.gif` (head + shoulders, 336×220) — **per-frame face-aligned**: each frame's top-most opaque y is found and the crop starts there. Klaus's face stays in the same spot whether he's resting or jumping, so it never bounces out of an avatar circle.

Backdrop sampled from pixel `(0,0)` of frame 1, matched with `BG_TOLERANCE = 18`; padding `8`.

Both GIFs ship as `NSDataAsset`s (not imagesets) because SwiftUI's `Image` doesn't play animated GIFs. `Contents.json` declares UTI `com.compuserve.gif`.

**`KlausMascotView`** has three pieces:

1. `DisplayMode` (`.full` / `.portrait`). `.full` aspect-fits into a `size × size` square (preserves tall aspect ratio). `.portrait` aspect-fills (square avatars get full coverage).
2. `KlausMascotAssets` — singleton loader. Decodes each GIF once via `CGImageSourceCreateWithData`, extracts every frame as `UIImage`, reads delays from `kCGImagePropertyGIFDictionary` (`UnclampedDelayTime` preferred, clamped to 0.02 s minimum). Cache guarded by `NSLock`.
3. `KlausAnimatedImage` (`UIViewRepresentable`) — wraps `UIImageView` with `animationImages` / `animationDuration` / `animationRepeatCount = 0`. Layer filters set to `.nearest` for crisp pixel-art. **Implements `sizeThatFits(_:uiView:context:)` (returning the proposed size), sets `clipsToBounds = true`, and drops `contentHugging` / `contentCompressionResistance` to `.defaultLow - 1` on both axes.** Without these, SwiftUI consults the intrinsic content size (336×446 *points* at scale 1) and the mascot renders at hundreds of points tall in a 44-pt frame.

**Where Klaus appears:** `WiFiAssistantView` header (44 pt, `.portrait`, circle), each assistant chat bubble avatar (34 pt `.portrait`), Signal-tab "Chat with Klaus" CTA (46 pt `.portrait`), `PaywallView` page 3 (`.portrait`), `AboutView` Klaus intro, and as the signing character on every report card across the app — `DashboardView.speedReport` ("Klaus's Wi-Fi Report"), `SurveyInsightsView.gradeHeader` ("KLAUS'S INSIGHTS") + `SurveyInsightsPlaceholder` ("Klaus needs more data") + `LockedInsightsCard` ("Klaus's Full Report"), and `DeviceDiscoveryView` (header replaces the app logo with a 56 pt mascot, "Klaus's Security Report" replaces "Network Assessment", "Klaus's Tips to Lock It Down" replaces "Protect Your Network", per-device "Klaus's Tips: Is This Yours?" identification card). The summary copy on each surface uses Klaus's first-person voice ("Beep boop — your connection is…", "I identified all 12 devices…") so the same character users meet in the chat sheet is the one signing every analytical output.

**Personality (copy only):** Greeting "Beep boop — hi there! I'm Klaus…", fallback "Hmm, my little antenna didn't quite pick that one up…", 14 thinking phrases ("Booting up my brain", "Recalibrating my dish", etc.). The 24 curated Q&A answers themselves are unchanged.

### WiFi Buddy Assistant (pseudo-AI chat)

Full-screen chat sheet from the Signal tab's "Chat with Klaus" CTA. Entirely offline.

`WiFiAssistantView.swift` has four pieces:

1. **Knowledge base (`WiFiAssistantKnowledge.entries`)** — 24 `AssistantQA` structs across Coverage / Reliability / Setup / Streaming / Security / Speed / Gaming. Each cross-references the relevant tab where applicable.
2. **Matching engine (`WiFiAssistantEngine`)** — `sanitize` (strips control chars, caps at 300 chars), `tokenize` (lowercase, `CharacterSet.alphanumerics.inverted`, drops a ~40-entry English stopword set), `findBestAnswer(for:in:)` (counts keyword matches per QA, returns highest score ≥ 1), `relatedQuestions(for:count:)`, `fallbackSuggestions(count:)`, `starterQuestions`.

## Device discovery & classification

`NetworkScanner` runs eight identification layers in this order:

1. **Bonjour** (19 service types including `_afpovertcp._tcp`, `_rfb._tcp`, `_apple-mobdev2._tcp`, `_touch-able._tcp`, `_amzn-wplay._tcp`). Long-lived browsers run for the entire scan duration (was 3 s; now 10+ s) so iOS Local Network permission delays don't drop the first-scan results. Resolution uses `BonjourResolver` (Foundation `NetService`) — returns hostname + IPv4 in a single callback without needing a TCP handshake. `BonjourCollector` is `@unchecked Sendable` and bridges `NWBrowser` background callbacks into the `@MainActor` scanner. `applyBonjourMetadataToExistingDevices` backfills names onto port-scan-discovered devices.
2. **SSDP / UPnP** — UDP M-SEARCH multicast to `239.255.255.250:1900` with three ST values (`ssdp:all`, `upnp:rootdevice`, `urn:dial-multiscreen-org:service:dial:1`) 100 ms apart. Parses HTTP-style headers (SERVER / ST / USN / LOCATION).
3. **UPnP device descriptions** — fetches LOCATION URLs for `<friendlyName>` / `<manufacturer>` / `<modelName>`. Critical for Echo, etc., whose generic SERVER headers contain no brand info.
4. **TCP port fingerprinting** — 21 ports per host (80, 443, 62078, 548, 445, 7000, 8080, 9100, 631, 554, 8008, 8443, 1883, 3689, 22, 139, 5353, 8888, 5000, 515, 3000). Batch size **10** (was 20) with **800 ms** timeout — tradeoff between iOS connection caps and missing slow hosts.
5. **TCP liveness** — for hosts with no probed ports open, secondary probe to 443/80/7/1; fast TCP RST < 500 ms means alive (firewalled host that drops specific probes).
6. **Reverse DNS** — `getnameinfo`. `cleanHostname()` strips `.local`, `.home`, `.lan`, etc. Strong name-based matches (gameConsole, computer, phone, printer, speaker) override weaker port-based classifications (`.iotDevice`, `.smartTV`).
7. **HTTP fingerprinting** — for devices still missing a name / classified `.unknown` / lacking manufacturer, parallel async HTTP GETs to port 80/8080. Analyzes Server header, HTML `<title>`, body keywords, and falls back to known UPnP description paths (`/xml/device_description.xml`, `/rootDesc.xml`, `/description.xml`).
8. **MAC + OUI** — `MACAddressResolver` reads kernel ARP via `sysctl(NET_RT_FLAGS)`. `OUIDatabase.manufacturer(forMAC:)` maps the OUI to a vendor (~300 entries). Locally-administered MACs (privacy-randomized) skip the OUI lookup. `refineClassificationFromVendor` upgrades `.unknown` / `.iotDevice` when vendor + ports give a strong signal.

### Classification fixes (record kept for reference)

- **AirPlay + computer co-detection** — Mac Mini / MacBook Pro were classified `.smartTV` because AirPlay was checked before File Sharing / port evidence. The classifier now checks for co-existing computer indicators (`_afpovertcp._tcp`, `_rfb._tcp`, `_smb._tcp`, ports 548/445/22, computer keywords) before defaulting to `.smartTV`.
- **Port-based SMB + AirPlay** — `classifyByPorts` previously excluded `.computer` when 7000 (AirPlay) + 445 (SMB) coexisted. Macs commonly have both. SMB now implies `.computer` regardless of AirPlay.
- **Nintendo Switch** — name-based reclassification only overrode `.unknown`. Now strong name matches also override weaker port-based classifications.

### Display naming

Priority: `customName` (if device is trusted and renamed) → `isCurrentDevice` / router → Bonjour name → cleaned hostname → `<OUI vendor> <ShortName>` → `<manufacturer> <ShortName>` → bare `deviceType.rawValue`. Web-only-port devices (only 80/443/8080/8888/8443/3000/5000) classify as `.iotDevice` instead of `.unknown`.

### Custom device names

`DiscoveredDevice.customName` is set via `NetworkScanner.setCustomName(_:name:)`, which sanitizes (strip control/newlines, collapse whitespace, trim, cap at 40 chars) and persists to `UserDefaults` under the **current network's ID**. Only allowed on **trusted** devices (untrusted rename would give false sense of recognition). Un-trusting clears the nickname.

UI: detail sheet shows a "Name This Device" / "Rename Device" button below Trust (only when `device.isTrusted`). Alert with `TextField` (word capitalization, 24/40-char cap, autocorrection off). Actions: Save / Clear Name (destructive, only when a name exists) / Cancel.

### Per-network trust + nickname scoping

Trust flags and nicknames are scoped by **gateway MAC address** because home routers default to `192.168.1.x` / `192.168.0.x`, so flat IP-keyed storage caused a stranger's device on a different network to silently inherit a "TRUSTED" badge.

- Storage keys: `trustedDevicesByNetwork: [networkID: [trustedIP]]`, `customDeviceNamesByNetwork: [networkID: [ip: nickname]]`.
- `networkID = "gw:<lowercased-colon-mac>"` from the ARP table. Returns `nil` (and treats every device as untrusted) when the gateway MAC isn't resolvable or has the locally-administered bit set.
- `applyTrustAndCustomNames` runs as a `defer` inside `resolveMACAddresses`, so trust flags appear at the end of every scan.
- `setTrusted` / `setCustomName` are no-ops when `currentNetworkID == nil`.
- **One-shot legacy migration** (`migrateLegacyTrustIfNeeded`, gated by `trustMigrationCompleted_v2`): pre-upgrade flat data is carried forward only when **at least half** of the legacy trusted IPs are present on the current network. Otherwise legacy entries are dropped — better to untrust everything than to silently re-trust strangers.

### New-device alerts

`NetworkScanner.detectNewDeviceArrivals(networkID:)` runs after `applyTrustAndCustomNames`. Maintains `seenMACsByNetwork` baseline in `UserDefaults`. Skips networks with 0 trusted IPs (don't alert on coffee-shop Wi-Fi). First-ever scan only populates the baseline. Subsequent scans diff and forward unseen MACs to `NewDeviceAlertNotifier.postNewDeviceAlerts(_:)`. Randomized MACs are excluded — they'd look "new" every few weeks.

### Randomized-MAC wording

When `hasRandomizedMAC == true` we now show the plain `deviceType.rawValue` (e.g., "Phone / Tablet"), not the prefixed "Private X" label. Subtitle reads **"Uses Private Wi-Fi Address"** — the exact phrase iOS Settings uses, which reframes it as a normal privacy feature rather than something suspicious. MAC row label is `(Private)`. Tips card explicitly tells the user it's almost always one of their own devices.

This UX fix compounds with the `NSBonjourServices` Info.plist requirement: before that fix Bonjour browsing failed entirely (`NoAuth(-65555)`) and **every** Apple device on the network collapsed into the randomized-MAC fallback, so users saw a screen full of "Private X" devices.

## Live topology card (Speed tab)

## Submission gotchas (App Review history)

The build went through three rejection rounds before the first approval. The durable lessons:

### Free trial UI must match StoreKit reality

Round-2 rejection (2.1(b)): `PaywallView` only surfaced trial copy when Yearly was selected, but `Configuration.storekit` declared the same intro offer on Monthly. Reviewer selecting Monthly saw a "Subscribe" CTA but the system StoreKit sheet would have started a free trial anyway → UI/IAP mismatch.

**Fix:** `buyButtonLabel` returns `"Start 3-Day Free Trial"` whenever `trialAvailableForSelection` is true, regardless of plan. The `selectedPlan == .yearly` guard was removed across `trialTimeline`, `disclosureLine`, `monthlyCardSubtitle`, etc. `selectedPlanDisplayPrice` helper centralizes the "live `displayPrice` with hardcoded fallback" logic for whichever plan is currently selected, so timeline + disclosure stay in lockstep on price/period changes.

### Surface real StoreKit failures

Round-3 rejection (2.1(b)): "An error appears when we tap on 'Start 3-Day Free Trial'." Pre-round-3, the catch in `ProStore.purchase(_:)` collapsed every failure mode (productUnavailable, purchaseNotAllowed, ineligibleForOffer, invalidOfferIdentifier, missing offer parameters, network, system, storefront, etc.) into a single `"Purchase failed. Please try again."` alert.

**Fix:** `purchaseErrorMessage(for:)` switches on `Product.PurchaseError` and `StoreKitError`, falls through to a typed `"\(type(of: error)): \(localizedDescription)"` capture for unknown subclasses. Same idea applied to `loadProducts()`'s catch. The diagnostic suffix sits in parentheses after the recovery hint — acceptable in shipping copy because users have already tapped Buy by then, and the win on review screenshots and support emails far outweighs the readability cost.

### Subscriptions must be attached to the build under review

The most common cause of "Buy / Restore returned no products" review failures: a subscription can be "Approved" in App Store Connect yet **not attached to a specific version review**. Reviewer's sandbox only trusts attached-to-the-binary. After every SKU rebuild, manually go to App Store Connect → My Apps → version → "In-App Purchases and Subscriptions" and attach both subscriptions.

Other ASC items to verify before resubmitting (in order of likelihood):

1. Both IAPs attached to the version (above).
2. Tax Category set on both subscriptions.
3. Subscription Group has at least one localization.
4. Each subscription has Display Name + Description for at least US English.
5. Introductory Offer status — the offer can be "Ready to Submit" independent of the parent subscription. Confirm both intro offers say "Ready to Submit" *and* are listed under the version's IAP section.
6. Pricing/availability for the reviewer's storefront (typically US).
7. Subscription Review Information — Apple requires a screenshot per subscription product.
8. Apple Developer Program License Agreement up to date (separate from Paid Apps Agreement).

### Deleted product IDs are permanent

After an ASC subscription rebuild (legal/tax/group-localization metadata drift forced the SKUs to be deleted and recreated), **Apple permanently blocks reuse of deleted product identifiers**. This is why the IDs carry the `.sub.` infix:

- Before: `com.wifibuddy.pro.monthly` / `com.wifibuddy.pro.yearly` (deleted).
- After: `com.wifibuddy.pro.sub.monthly` / `com.wifibuddy.pro.sub.yearly` (permanent).

When rotating SKUs, the literal strings live in **two files** that must agree (plus README + this doc):

- `SignalStrengthPainter/ProStore.swift` — `monthlyProductID` / `yearlyProductID` constants.
- `Configuration.storekit` — `productID` fields under each `subscriptions` entry. Also rotate all four `internalID` UUIDs (subscription + intro-offer per plan) so the local catalog treats them as fresh SKUs and doesn't carry over cached transaction state.

### Version trains close after submission

ASC errors after a SKU rebuild:

- **`90186 — Invalid Pre-Release Train. The train version '1.0' is closed for new build submissions`** — once a version is approved (or even just submitted to review without proper withdrawal), ASC locks that train and refuses any further uploads claiming the same `CFBundleShortVersionString`. Only fix: ship a higher version.
- **`90062 — CFBundleShortVersionString must contain a higher version than that of the previously approved version`** — same root cause.

Versioning convention going forward:

- **Resubmission / bug fix / SKU rebuild** → bump patch (`1.0.1 → 1.0.2`).
- **New feature** → bump minor (`1.0.x → 1.1.0`).
- **Major redesign** → bump major (`1.x → 2.0.0`).
- **Re-archive of the same version** (metadata fix without binary change, or rejected upload re-uploaded) → keep version, bump build only (`1.0.1 (2) → 1.0.1 (3)`). Even a rejected upload burns a build number.

### Info.plist literal-vs-substitution drift

The version has to be bumped in **four spots** to stay consistent because `GENERATE_INFOPLIST_FILE = NO` and the static `Info.plist` uses literal strings (`<string>1.0.1</string>`) instead of `$(MARKETING_VERSION)` / `$(CURRENT_PROJECT_VERSION)` substitutions:

- `SignalStrengthPainter/Info.plist`: `CFBundleShortVersionString`, `CFBundleVersion`.
- `SignalStrengthPainter.xcodeproj/project.pbxproj` (Debug + Release): `MARKETING_VERSION`, `CURRENT_PROJECT_VERSION`.

Why all four matter despite App Review only quoting Info.plist: Xcode's General tab reads the build settings, `agvtool` writes the build settings, and TestFlight's "What's New" string is sourced from the build settings on some upload paths. Leaving them stale silently causes drift across UI surfaces.

**Future improvement (not done):** convert plist entries to `$(MARKETING_VERSION)` / `$(CURRENT_PROJECT_VERSION)` substitutions so future bumps only require editing one place. Skipped during the resubmission crunch — botching the substitution would re-trigger `90062` under a different code.

## Swift 6 / strict-concurrency notes

Reference for not re-introducing patterns that the warning sweep already cleared:

- `OUIDatabase.entries` is built from a `rawEntries: [(String, String)]` tuple array via `Dictionary(rawEntries, uniquingKeysWith: { first, _ in first })` so duplicate OUIs (one MAC prefix appearing under multiple vendors) don't trigger duplicate-key warnings. Adding more entries is safe; duplicates silently defer to whoever was listed first.
- `SignalMapViewModel.startPingLoop` — `Timer.scheduledTimer` and `LatencyProbe`'s completion callback are nonisolated. Both are wrapped in `Task { @MainActor [weak self] in … }` to satisfy the concurrency checker.
- `ProStore.monthlyProductID` / `yearlyProductID` / `allProductIDs` are `nonisolated static let` so SwiftUI views and `@ViewBuilder` bodies can reference them without a main-actor hop.
- `DashboardView` uses the two-parameter `.onChange(of:perform:)` form (`{ _, newPhase in }`) — single-parameter form was deprecated in iOS 17.
- `NetworkScanner.probeLivenessRefused` and `probePort` use a `private final class AtomicBool: @unchecked Sendable` one-shot latch around an `NSLock`. The inner `finish` helpers are marked `@Sendable` because they're called from two different thread contexts (NWConnection state handler + `probeQueue.asyncAfter` timeout).
- `SWIFT_STRICT_CONCURRENCY` is still `minimal`. Bumping to `complete` / `targeted` would surface more Timer/DispatchQueue closure patterns (`SignalMapViewModel`, `NetworkTopologyMonitor` Combine `sink`s) that need similar `Task { @MainActor in … }` wrappers.
